**Reading the numbers.** Wall time is measured under 16-way concurrency, so it reflects *throughput under load*, not isolated single-event latency (a lone event runs faster -- e.g. PDHD imaging is ~90 s solo). **Peak RSS is per-process** and is independent of concurrency, so the memory columns are directly meaningful for capacity planning. RSS is the high-water mark (`VmHWM`) of that stage's wire-cell process.

All cells below are **min / median / mean / max**.

## Observations

- **Imaging dominates wall time**; clustering is typically 2-4x faster per event.
- **Typical memory is modest but the tail is heavy.** Median RSS is ~743 MB (imaging) and ~933 MB (clustering). A small number of events spike far higher: peak imaging RSS hit 9.9 GB (PDHD run027305 evt0) and peak clustering RSS hit 3.2 GB (PDHD run027305 evt0). The outliers cluster in **run027305** and **run028084** (busy / incomplete-readout events), driving the high `mean`/`max` versus `median`.
- **Concurrency headroom is large.** At 16-way with ~1 GB typical per stage the working set is ~16 GB; even a worst-case coincidence of the ~10 GB outliers is comfortable in 251 GiB. Memory is not the bottleneck at this concurrency.
- **PDVD imaging is slower than PDHD** (8 anodes vs 4, larger CRP planes) but comparable in memory; PDVD clustering time/memory is similar to PDHD.
